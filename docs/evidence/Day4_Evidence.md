# Day4 证据卡——发布表单与数据新增

## 一、表单字段设计说明

### 1. 通用字段设计
发布页面采用"类型切换表单"模式，所有业务类型共享以下通用字段：
- **标题**（必填）：作为信息的核心标识，便于用户快速了解内容
- **描述**（必填）：详细说明信息内容，帮助用户做出决策
- **图片**（选填）：提供视觉参考，增强信息可信度

### 2. 二手交易专属字段
- **价格**（必填）：二手商品的核心属性，用户最关心的信息之一
- **原价**（选填）：便于用户了解折扣力度
- **分类**（必填）：帮助用户快速筛选商品
- **成色**（选填）：描述商品新旧程度，影响用户购买决策

### 3. 失物招领专属字段
- **类型**（必填）：区分"我丢了"和"我捡到"两种场景
- **物品名称**（必填）：明确物品信息，便于失主识别
- **位置**（必填）：丢失或捡拾地点，是找回物品的关键线索
- **时间**（必填）：记录时间，帮助匹配相关信息

### 4. 拼单搭子专属字段
- **拼单价**（必填）：参与拼单的价格，是用户决策的重要依据
- **原价**（选填）：展示拼单优惠力度
- **成团人数**（必填）：明确拼单目标，帮助用户判断是否参与
- **截止时间**（必填）：限定拼单有效期，营造紧迫感

### 5. 跑腿委托专属字段
- **悬赏金额**（必填）：激励跑腿者接单的核心因素
- **起点位置**（必填）：任务起点，帮助跑腿者评估任务
- **终点位置**（必填）：任务终点，计算跑腿距离和难度
- **要求时间**（选填）：用户期望完成时间

## 二、表单校验规则设计

### 1. 通用校验规则
- **标题校验**：非空校验，确保信息有明确标识
- **描述校验**：非空校验，确保信息完整

### 2. 二手交易校验规则
- **价格校验**：非空且大于0，防止无效价格输入
- **分类校验**：必须选择分类，确保数据规范

### 3. 失物招领校验规则
- **物品名称校验**：非空校验
- **位置校验**：非空校验
- **时间校验**：非空校验，确保时间信息完整

### 4. 拼单搭子校验规则
- **拼单价校验**：非空且大于0
- **成团人数校验**：非空且大于0，确保合理的成团目标
- **截止时间校验**：非空校验，确保拼单有明确期限

### 5. 跑腿委托校验规则
- **悬赏金额校验**：非空且大于0，确保有足够激励
- **起点位置校验**：非空校验，确保任务可执行
- **终点位置校验**：非空校验，确保任务完整

## 三、数据新增实现

### 1. API接口设计
在 `src/api/` 目录下为四类业务分别封装了POST方法：
- `trade.ts`：`createTrade()` - 提交二手交易数据
- `lostFound.ts`：`createLostFound()` - 提交失物招领数据
- `groupBuy.ts`：`createGroupBuy()` - 提交拼单搭子数据
- `errand.ts`：`createErrand()` - 提交跑腿委托数据

### 2. 数据提交流程
1. 用户填写表单并点击"发布"按钮
2. 触发 `handleSubmit()` 函数
3. 先执行 `validateForm()` 进行表单校验
4. 校验通过后根据发布类型调用对应API
5. 使用Axios向JSON Server发送POST请求
6. 提交成功后跳转到对应列表页面
7. 提交失败时显示错误提示

### 3. 数据字段映射
提交的数据字段与 `db.json` 中的数据集合保持一致，确保列表页面能够正确渲染新增数据。
