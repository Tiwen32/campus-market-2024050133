## 2. 发布类型与字段设计

| 发布类型 | 对应数据集合 | 关键字段 | 设计理由 |
|---|---|---|---|
| 二手交易 | trades | title、category、price、condition、location、description | 需要展示商品基本信息和交易条件，帮助买家了解商品情况并做出购买决策 |
| 失物招领 | lostFounds | title、type、itemName、location、eventTime、description | 需要区分寻物和招领两种场景，并详细描述物品特征和丢失/捡拾信息，便于失主识别 |
| 拼单搭子 | groupBuys | title、type、targetCount、deadline、location、description | 需要说明拼单目标人数和截止时间，让参与者了解拼单进度和时间限制 |
| 跑腿委托 | errands | title、taskType、reward、from、to、deadline、description | 需要说明任务内容、起点终点位置和酬劳金额，让跑腿者评估任务难度和收益 |

## 3. 表单校验规则

为确保数据质量和用户体验，设置了以下表单校验规则：

-  标题不能为空 ：所有发布类型都需要标题作为信息标识；
-  描述不能为空 ：详细描述有助于其他用户了解信息内容；
-  二手交易价格必须大于 0 ：价格是商品交易的核心属性，不能为0或负数；
-  二手交易分类必须选择 ：分类便于商品筛选和浏览；
-  失物招领物品名称不能为空 ：明确物品名称是找回失物的关键；
-  失物招领位置不能为空 ：位置信息有助于定位丢失或捡拾地点；
-  失物招领时间不能为空 ：时间信息帮助匹配相关线索；
-  拼单搭子拼单价必须大于 0 ：价格是用户参与拼单的重要参考；
-  拼单搭子成团人数必须大于 0 ：合理的成团目标确保拼单可行性；
-  拼单搭子截止时间不能为空 ：明确的截止时间营造紧迫感；
-  跑腿委托悬赏金额必须大于 0 ：足够的酬劳是激励跑腿者接单的关键；
-  跑腿委托起点位置不能为空 ：明确起点确保任务可执行；
-  跑腿委托终点位置不能为空 ：明确终点完整描述任务范围。

## 7. 遇到的问题与解决方法

 问题：表单提交后页面刷新 

-  现象 ：最初测试时，点击"发布"按钮后页面会重新刷新，导致输入内容丢失；
-  原因 ：表单使用了默认的提交行为，没有阻止表单的默认提交事件；
-  解决方法 ：在 `<form>` 标签上添加 `@submit.prevent` 修饰符，阻止表单默认提交行为，改为通过 JavaScript 函数处理提交逻辑。

 问题：POST 请求失败 

-  现象 ：提交表单时控制台显示网络请求失败；
-  原因 ：JSON Server 服务未启动，导致请求无法到达；
-  解决方法 ：先启动 `npm run mock` 命令启动 Mock 服务，确保 `http://localhost:3001` 可以正常访问，然后再进行表单提交测试。

## 8. 今日反思

 发布表单 、 表单校验 和 数据新增 是完整 Web 应用不可或缺的核心功能。发布表单是用户与系统交互的入口，它负责收集用户输入的业务数据；表单校验则是数据质量的第一道防线，通过对用户输入的验证，确保提交的数据符合业务规则，避免无效数据进入系统；数据新增则是将用户输入转化为持久化数据的关键环节，它使得用户的操作能够产生实际效果，在列表页面中展示出来。

这三个环节构成了一个完整的用户交互闭环：用户通过发布表单输入信息，系统通过表单校验确保数据有效，然后通过数据新增将信息保存到后端，最终在列表页面展示给其他用户。缺少任何一个环节，Web 应用都无法正常运转。发布表单设计是否合理、表单校验是否完善、数据新增是否可靠，直接影响用户体验和系统稳定性。在实现过程中，需要特别注意字段设计的业务贴合度、校验规则的合理性以及接口调用的可靠性，确保整个流程顺畅、数据准确。
